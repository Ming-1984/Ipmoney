# Ipmoney 专利交易小程序需求文档（优化补充版）

> 目标：让搜索列表的默认排序更贴近用户兴趣与区域产业特色；同时保持可控、可解释、可调参。
> 本文在现有需求基础上补充：业务边界、关键流程、检索字段规范、专利类型定义标准（发明/实用新型/外观设计）等。

## 0. 术语与范围（补充）

- **平台定位：** 居间/撮合服务，不直接替代专利权属转移的法定流程（过户/备案仍需按主管部门要求办理）。
- **品牌与视觉：** 前端主色建议为**橙色**（寓意成功/成交）；平台理念为“**专利点金台**”（把专利价值转化为可交易、可量化的收益）。
- **多端形态：** P0 支持 **微信小程序（买家/卖家） + 用户 H5（Taro；电脑端可用） + PC Web 管理后台（运营/客服/财务）**。
  - **电脑端 H5（P0）：** 支持浏览/搜索/详情/收藏/咨询；**下单与支付引导回小程序**（二维码/链接），避免 H5 支付/合规与渠道成本。
- **交易对象：** 专利交易为主；新增**书画专区**（书画作品交易，走订金+尾款）；产学研需求/成果为展示与咨询。
- **交易方式：** 专利以转让为主、许可为辅（P0 支持许可：独占/排他/普通；前台默认转让）；书画专区以**直接交易**为主（拍卖/委托为 P1）。
- **沉睡专利：** 转让次数为 0 的专利（`transferCount=0`），卡片标签展示“沉睡专利”。
- **高新退役专利：** 高新技术企业曾经使用过、已退役/闲置的优质专利；**由企业提交材料，平台审核通过后标记**（非仅凭转让次数判断）。
- **产业集群特色专利：** 以**产业集群**为核心维度组织的专题专利集合；以 `clusterId` 独立字段标识（与行政区 `regionCode` 解耦）。
- **开放许可：** 交易方式为**许可**的专利（`tradeMode=LICENSE`），用于“开放许可”专题入口。
- **专利公告：** 公告/挂牌清单等（如“广东工业大学专利开发许可清单(202601)”），支持列表+详情（**不提供附件功能**）。
- **订金：** 用于表达交易意向并触发客服介入；是否可退、退还条件、违约责任必须在协议与流程中明确。
- **上架信息来源：** 用户发布为主（P0 以用户上传/后台录入为准）；P1 可选接入官方/第三方检索源做自动回填与校验（减少手填、降低造假），书画专区同理走外部适配层。
- **数据地图/大数据分析（P1 预留）：** 数据地图包含专利地图（发明/实用新型/外观设计）+书画成果地图+技术经理人地图/科学家地图；大数据分析包含**交易库/成果库/产学研库**，先预留架构与后台入口。

## 1. 角色与权限（建议补充）

- **游客：** 可浏览公开信息与搜索；敏感字段（联系方式、证件等）默认不展示。
- **已注册未认证：** 可收藏/咨询；发布与支付前需完成实名认证/主体认证。
- **身份/主体认证（P0）：**
  - **个人：** 可直接授权信息完成注册（无需后台审核）；发布/交易前如需更严格实名可作为 P1 追加。
  - **企业：** 提交主体信息与材料，后台审核通过后可发布/交易；**通过后可在小程序“机构展示”中被检索/展示**。
  - **科研院校：** 提交主体信息与材料，后台审核通过后可发布/交易；**通过后可在小程序“机构展示”中被检索/展示**。
  - **政府：** 提交主体信息与材料，后台审核通过后可发布/交易（是否在前台展示可配置）。
  - **行业协会/学会：** 提交主体信息与材料，后台审核通过后可发布/交易（是否在前台展示可配置）。
  - **技术经理人：** 提交个人资质信息与材料，后台审核通过后可发布/交易（可作为“服务方/撮合方”展示）。
- **平台运营：** 内容审核、用户认证审核、类目/标签/敏感词配置。
- **客服：** 跟单、聊天监管、线下流程记录、发起退款/争议处理流转。
- **财务：** 对账、退款、佣金结算、发票。

## 2. 核心业务流程（端到端补充）

1. **卖家认证 → 发布专利/书画商品 → 平台审核 → 上架**
2. **买家搜索/筛选 → 在线咨询 → 支付订金 → 系统分配客服**
3. **客服介入：** 资料核验/尽调清单 → 线下签约（转让/许可合同）→ 权属变更/备案（如适用）
4. **尾款与结算：** 线下合同签署确认 → 线上支付尾款（平台托管）→ 权属变更完成 → 结算佣金并放款 → 订单完成/归档
5. **退款/争议：** 按“可退/不可退”规则、证据上传、客服/运营处理

## 3. 检索功能与专利类型分类标准（用于搜索/筛选）

### 3.1 三种专利类型的法定定义（《专利法》第二条）

- **发明：** 对产品、方法或者其改进所提出的新的技术方案。
- **实用新型：** 对产品的形状、构造或者其结合所提出的适于实用的新的技术方案。
- **外观设计：** 对产品的整体或者局部的形状、图案或者其结合以及色彩与形状、图案的结合所作出的富有美感并适于工业应用的新设计。

### 3.2 存续期提示（《专利法》第四十二条）

- **发明：** 20 年；**实用新型：** 10 年；**外观设计：** 15 年（均自申请日起算）。

### 3.3 检索字段规范（建议最小集合）

- **关键标识：** 专利号/申请号/公开号（支持多格式输入，统一规范化存储）。
- **专利类型：** 发明/实用新型/外观设计（必填，支持自动识别后可改）。
- **分类号：** 发明/实用新型用 IPC；外观设计用 Locarno（LOC）分类（用于“行业/领域”精细筛选）。
- **法律状态：** 申请中、已授权、终止/失效、无效（以数据源为准）。
- **权利人/申请人、发明人/设计人、申请日、公开/公告日**。
- **交易字段：** 交易方式（转让/许可）、标价/面议、订金、地区、行业标签、转让次数、发布时间、更新时间。

### 3.4 类型自动识别（可选增强）

- **若输入为公开号/公告号：** 末尾常见标记 A/B → 发明，U → 实用新型，S → 外观设计（以各国文献种类代码为准，存在例外需人工确认）。
- **若输入为中国申请号：** 常见格式 `YYYYTxxxxxxx.x`，其中 `T` 通常为 1/2/3 分别对应发明/实用新型/外观设计（PCT 等特殊号段需额外规则）。

### 3.6 专利号输入正则与规范化规则（P0 必做）

> 目标：让搜索列表的默认排序更贴近用户兴趣与区域产业特色；同时保持可控、可解释、可调参。

#### 3.6.1 支持的输入类型（P0：CN）

- **中国申请号（Application No.）**
  - **较新常见格式：** `YYYYTXXXXXXX.C`（含点号）或 `YYYYTXXXXXXXC`（纯数字）
    - `YYYY`=申请年（4 位），`T`=类型位（`1`/`2`/`3`），`XXXXXXX`=流水号（7 位），`C`=校验/控制位（1 位数字）
    - 例：`2023113409720` / `202311340972.0`
  - **历史常见格式（兼容）：** `YYTXXXXX.C`（含点号）或 `YYTXXXXX C`（纯数字/含分隔符）
    - `YY`=申请年（2 位），`T`=类型位（`1`/`2`/`3`），`XXXXX`=流水号（常见 5 位），`C`=校验/控制位（1 位数字）
    - 例（示意）：`96123456.1`
- **中国专利号（Patent No.）**：常见写法为 `ZL` + 申请号（本质上仍以申请号为主键）。
  - 例：`ZL202311340972.0`
- **中国公开号/公告号（Publication No. / Grant No.）**：`CN` + 数字 + **文献种类代码（Kind code）**
  - 例：`CN114123456A`、`CN114123456B`、`CN214123456U`、`CN314123456S`（Kind code 也可能包含数字，如 `A1/A8` 等，按数据源为准）

#### 3.6.2 规范化存储（建议输出字段）

- `jurisdiction`：国家/地区代码（WIPO ST.3），P0 默认 `CN`
- `identifier_type`：`CN_APPLICATION` / `CN_PUBLICATION` / `CN_PATENT(ZL)`
- `application_no_norm`：去掉点号与前缀后的纯数字（CN 常见长度 9 或 13 位），如 `2023113409720`
- `application_no_display`：展示格式（默认在末位前插入点号），如 `202311340972.0`
- `publication_no_norm`：去空格、全大写，如 `CN114123456A8`
- `patent_type`：`INVENTION` / `UTILITY_MODEL` / `DESIGN`
- `kind_code`：如 `A`/`B`/`U`/`S`/`A1`/`A8`…
- `parse_status`：`OK` / `FORMAT_ERROR` / `AMBIGUOUS`

#### 3.6.3 规范化清洗流程（先清洗再匹配）

1. **trim**：去首尾空白
2. **统一字符集**：全角转半角（数字、字母、冒号、括号等）；字母转大写
3. **删除噪声前缀/标签**：如 `专利号`/`申请号`/`公开号`/`公告号`/`：` 等（保留核心内容）
4. **移除分隔符**：空格、`-`、`_`、中文顿号/逗号等；**保留点号 `.`**（用于识别申请号校验位）
5. **模式匹配**：按优先级尝试 `CN_APPLICATION` → `CN_PUBLICATION` → 其他（P1 扩展）
6. **二次规范化**：输出 `*_norm` 与 `*_display`，并填充 `patent_type/kind_code`

#### 3.6.4 正则规则（建议“宽松识别 + 严格校验分层”）

> 说明：小程序/前端建议用“宽松识别”提升容错；后端入库前再做“严格校验 + 外部数据源核验”。

- **CN 申请号/专利号（宽松识别，允许 CN/ZL 前缀、允许有无点号；兼容新旧格式）**

```regex
^(?:CN)?(?:ZL)?(?<year>19\\d{2}|20\\d{2})(?<type>[123])(?<serial>\\d{7})\\.?((?<check>\\d))$
```

```regex
^(?:CN)?(?:ZL)?(?<year>\\d{2})(?<type>[123])(?<serial>\\d{5})\\.?((?<check>\\d))$
```

- **CN 公开号/公告号（宽松识别，允许 7~9 位数字 + Kind code，Kind code 允许带 1 位数字）**

```regex
^(?:CN)?(?<number>\\d{7,9})(?<kind>[A-Z]\\d?)$
```

- **类型映射（CN）**
  - 申请号 `type=1/2/3` → `发明/实用新型/外观设计`
  - 公开号 `kind=U` → `实用新型`；`kind=S` → `外观设计`；`kind=A/B` 通常为 `发明`（以数据源为准）

#### 3.6.5 校验策略（P0/P1）

- **P0：** 仅做“结构校验”（位数、字符集、前缀、Kind code 形态）；不强制校验位计算，避免误判导致用户无法发布。
- **P1：** 若确定数据源（CNIPA/第三方）提供校验位算法或校验接口，可增加“校验位/权属一致性/法律状态一致性”校验，并对异常给出可解释提示（例如“号码格式正确但未能在数据源检索到”）。

### 3.7 书画专区分类与字段规范（联网调研）

> 目标：统一书画专区分类维度与发布字段，便于检索与后续对接外部检索源。

- **一级类目：** 书法 / 绘画（书画专区主入口）。
- **书法书体（常用五体）：** 楷书、行书、草书、隶书、篆书（以行业通行分类为准）。
- **国画题材（三大类）：** 人物画、山水画、花鸟画（可扩展“其他”）。
- **作品要素（P0 必填）：** 作品名称、创作日期、创作者、作品介绍、著作权登记证书编号、作品图片（≥1）。
- **作品要素（P0 可选）：** 尺寸/材质、书体/题材、地区、标价/面议、订金、权属材料附件。
- **外部检索口径（P1 可选）：** 以著作权登记证书编号 + 作品名称/作者为主，接入外部检索源后返回“登记信息摘要+来源链接”；P0 先走后台人工核验并记录结果。

### 4.2 数据库字段设计（核心表，关系型数据库）

> 目标：让搜索列表的默认排序更贴近用户兴趣与区域产业特色；同时保持可控、可解释、可调参。

**A. 专利主数据（建议）**

- `patents`
  - `id`（PK）
  - `jurisdiction`（`CN`）
  - `application_no_norm`（唯一，建议与 `jurisdiction` 组成唯一键）
  - `application_no_display`
  - `patent_type`（`INVENTION/UTILITY_MODEL/DESIGN`）
  - `title`、`abstract`
  - `filing_date`、`publication_date`、`grant_date`（可空）
  - `legal_status`（归一化枚举）+ `legal_status_raw`（保留原始文本/代码）

  - `created_at`、`updated_at`

  - `id`（PK）、`patent_id`（FK）
  - `id_type`（`APPLICATION/PATENT(ZL)/PUBLICATION`）
  - `id_value_norm`（唯一，便于反查）

  - `date`（公开/公告日等，可空）

## 5. 数据来源与录入方案（已确认：用户上传 + 后台录入/导入）

> 本期不接入 CNIPA/第三方检索接口：专利信息以“用户发布时上传/填写 + 后台补录/导入”为准，并通过审核与材料校验控制风险。

### 5.1 本期数据来源

1. **用户上传（主）**
   - 发布专利商品时：输入专利号/申请号/公开号（按 3.6 规范化）+ 填写交易信息 + 上传权属/授权材料。
   - 发布书画作品时：填写作品信息 + 著作权登记证书编号 + 图片/权属材料（按 3.7 规范化）。
2. **后台录入/导入（辅）**
   - 运营可后台创建 `patents` 主数据、创建/代发 `listings`，并支持 Excel 批量导入（用于“精选交易/平台招商内容”）。
   - 书画专区支持后台录入/批量导入作品信息，用于冷启动与精选展示。

### 5.2 数据质量与风控（取代“外部权威核验”）

- **结构校验：** 专利号码/书画字段格式与类型映射（3.6/3.7），字段必填校验。
- **材料校验：** 上架前必须上传权属/授权链路材料（7.3），审核通过后方可展示“可交易”。

- **数据标识：** 专利/书画详情页展示“数据来源=用户上传/后台录入”与更新时间，避免用户误解为官方数据。

### 5.3 可选扩展（P1）：外部数据源接入

> 若后续需要“自动回填/法律状态自动刷新/权威核验”，再接入外部数据源，并通过适配层隔离差异。

3. **公开网页抓取/非公开接口（不建议生产）**：稳定性与合规风险高。

### 5.4 统一适配层（Provider Adapter）（P1）

- 抽象接口（示例）：
  - `lookupByApplicationNo(appNo_norm) -> PatentSnapshot`

## 7. 关键业务规则缺口（必须定稿，否则无法落库/联调）

### 7.1 订金与退款规则（已确认，按规范落地）

> 关键词：**订金线上支付、资金平台托管、线下签合同后再付尾款、退款有规则可追溯**。

**订金金额（可配置参数）**

- 平台支持 3 种策略：**固定值 / 阶梯 / 按标的总价比例（含上限/下限）**。
- **P0 默认值（后台可配置）：** 若为一口价（`priceType=FIXED`），订金按 `比例` 计算；若为面议（`priceType=NEGOTIABLE`），订金按 `固定值`。
  - `deposit_rate = 5%`
  - `deposit_min = ¥1,000`
  - `deposit_max = ¥20,000`
  - `deposit_fixed_for_negotiable = ¥2,000`
- **计算规则（P0）：**
  - `FIXED`：`deposit_amount = clamp(round(price_amount * deposit_rate), deposit_min, deposit_max)`（且不得超过 `price_amount`）
  - `NEGOTIABLE`：`deposit_amount = deposit_fixed_for_negotiable`

**资金流与抵扣**

- 订金由买家支付进入平台托管账户，**冻结**至“退款完成”或“交易完成放款”。
- 订金计入交易总价：尾款 = 成交价 - 订金（尾款同样线上支付并托管）。

**退款条件（按订单阶段）**

- **自动可退（系统秒退，P0 默认 30 分钟）：** 买家支付订金后 **30 分钟内**，且**客服未开始线下处理**时，买家可无理由申请退订金并系统原路秒退。
  - “未开始线下处理”定义（P0）：客服未点击“开始处理/开始跟单”，且工单未产生任何需双方配合的线下动作记录（如：未发起材料清单、未上传证据/合同页、未记录已寄送/已签署等）。
- **人工审核可退：** 已分配客服、但**线下合同未签署**时，满足以下任一条件可申请退订金：
  - 卖家未在规定时限内补齐权属/授权材料或材料明显不符
  - 双方协商一致取消（需双方线上确认或客服记录）
  - 平台风控/审核判定信息虚假或涉嫌侵权
- **原则不可退：** 线下合同已签署且无卖家违约/重大瑕疵情形时，买家单方反悔订金不退。
- **争议处理：** 发生纠纷（例如合同争议、权属争议）进入“争议工单”，由客服/运营按证据裁定（可升级法务）。

**退款流程（状态机）**

1. 买家发起退款 → 系统判断是否满足“自动可退”
2. 自动可退：直接原路退回并关闭订单
3. 人工审核：生成退款工单 → 客服收集证据/双方确认 → 运营审批 → 原路退款
4. 全链路记录：退款原因、证据附件、审批人、时间戳、支付平台退款单号（用于对账与审计）

### 7.2 平台收费与结算

**佣金（已确认：卖家承担）**

- 佣金从卖家应收款中扣除：卖家收款 =（订金 + 尾款）- 佣金。
- 佣金计费方式（可配置）：
  - **P0 默认：按成交价比例**（卖家承担），并提供下限/上限控制：`commission = clamp(round(deal_amount * 5%), ¥1,000, ¥50,000)`。
  - P1 可扩展：固定服务费 / 分档（可按类目/金额分段）。

**结算点（已确认：尾款线上支付）**

- 线下签署合同后，由客服/运营在后台完成“合同签署确认”里程碑，系统解锁买家“支付尾款”入口。
- 买家线上支付尾款进入平台托管账户；**P0 默认仅在“权属变更完成确认”后放款与结算**（超时自动放款默认关闭，仅支持人工特批）。
  - **放款方式（P0 默认）：** 财务线下放款（银行转账/企业网银等）→ 后台回传放款凭证与流水号 → 系统记账完成并关闭订单。

**发票（P0：线下开票，平台仅提供上传/下载）**

> 定位：平台仅对“平台服务费/佣金”开具发票；**不对专利转让/许可标的价款开票**（标的发票由卖家与买家线下另行约定）。

- **开票范围（P0 默认）：** 仅开具“平台服务费/佣金”发票；开票金额默认为 `commission_amount`（见结算台账）。
- **处理方式（P0）：** 卖家如需开票，线下联系平台（客服/财务）收集抬头信息 → 财务线下开票 → 后台选择**已完成订单**上传电子发票文件（仅 PDF）并可填写发票号码/开票日期 → 系统通知卖家在平台内下载。
- **系统不做（P0）：** 不做“开票申请单/状态流转/邮箱投递”；抬头与税号由财务线下收集（如需线上申请与专票作为 P1 扩展）。
- **交接提醒（重要）：** 当前实现已包含“开票申请/状态流转”相关接口（如 `POST /orders/:orderId/invoice-requests`、`GET /invoices`）用于前端流程对齐；后续后端真实实现时可沿用该契约，但需补齐持久化、权限与审计留痕。
- **留痕与权限（P0）：** 发票附件属于敏感资料；仅财务/管理员可上传/替换；下载需鉴权；上传/替换/下载写审计日志（who/when/what）。

### 7.3 线下签约/权属变更的“交付清单”

> 已确认：合同线下签署完成后，再回到平台支付尾款。

- 合同模板：转让/许可（独占/排他/普通）模板与必填条款清单（平台提供模板下载，线下签署后需回传关键页）。
- 权属材料：证书、权利人主体证明、授权/转让链路（如有）、质押/许可现状声明。
- 关键里程碑：合同签署、缴费、提交变更、变更完成；每一步的责任人与证据归档。

### 7.4 同一专利多次上架的规则

> 已确认：允许多个卖家上架同一 `application_no_norm`，由后台审核与风控把关。

- **上架策略：** 同一专利可存在多个有效 `listings`（不同卖家/不同价格/不同交易方式）。
- **审核要求：** 每个 listing 必须单独通过权属/授权材料审核；支持“疑似冒用”标记与拒绝上架。

- **举报与处置：** 买家可举报；客服生成工单并可临时下架；重复违规卖家可封禁并保留证据链。

##### 2.1 智能推荐算法 + 地域特色推荐（新增，P0）

> 目标：让搜索列表的默认排序更贴近用户兴趣与区域产业特色；同时保持可控、可解释、可调参。

**智能推荐（多维度权重，实时计算）**

- **推荐信号（P0）：**
  - 信息发布时间（新鲜度/时间衰减）
  - 详情点击量（`view_count`）
  - 收藏热度（`favorite_count`）
  - 咨询热度（`consult_count`，以“点击咨询/留资/联系客服”事件计）
- **推荐评分（示例公式，可配置权重）：**
  - `score = w_time*decay(age) + w_view*log1p(view) + w_fav*log1p(fav) + w_consult*log1p(consult) + w_region*regionMatch + w_featured*featuredBoost + w_user*tagSimilarity`
  - 说明：
    - `decay(age)`：时间衰减（如半衰期 `recommend_time_decay_half_life_hours`）
    - `log1p(x)`：抑制刷量（热度增长递减）
    - `tagSimilarity`：基于用户历史行为（收藏/咨询/成交）形成的“兴趣标签向量”，与信息的“产业标签”相似度（冷启动为 0）
- **实时性（P0 方案）：**
  - 统计字段在服务端增量更新（浏览/收藏/咨询事件落库），推荐排序在查询时按最新计数+当前权重实时计算（权重调整后立即生效）。
  - 反作弊（最低成本版）：对同一用户/设备对同一信息的浏览/咨询计数做时间窗去重（如 24h 只记 1 次）。

**地域特色推荐（LBS + 区域产业标签）**

- **区域优先：** 用户授权定位（LBS）或手动选择地区后，系统优先展示用户所在区域（同市 > 同省 > 其他）的专利成果/交易信息。
- **特色产业专利置顶：**
  - 后台支持把某条信息标记为“**省级/市级特色产业专利**”（并可设置置顶优先级/排序权重）。
  - 在首页推荐与搜索列表中对符合区域的“特色产业专利”给予高权重加成，必要时置顶展示（同时保留“为什么推荐/置顶”的解释文案）。
- **搜索与推荐融合：**
  - 搜索有关键词时：在文本相关性（relevance）基础上叠加 `score` 与 `region/featured` 加成，避免出现“相关但不本地/不热门”长期排前的问题。
- 无关键词（空搜索）时：默认按 `score` 输出推荐排序；首页展示最新专利。

##### 2.2 智能体与语音搜索（P1 预留）

> 目标：用户用自然语言（含语音）描述需求，系统自动拆解关键词与应用场景，输出专利/书画/需求/成果的匹配结果；保持可解释与可回溯。

- **入口（小程序优先）：** 首页/搜索页增加“IPMONEY 帮我找...”入口（语音或文本）。
- **端侧策略：** 小程序支持语音输入；H5 仅提示“到小程序使用语音搜索”。
- **能力范围：** 覆盖专利交易、书画专区、产学研需求、成果展示四类内容。
- **交互范式（成熟方案）：** 语音 → 实时转写 → 用户确认/编辑 → 结构化检索（类型/行业/地区/预算/合作方式等）。
- **结果展示：** 首屏显示“智能匹配卡片 + 推荐理由/匹配字段”，其余结果按常规排序列表展示。

##### 2.3 内容智能解析与评分闭环（P1 预留）

> 目标：让非专业用户能读懂专利/书画/需求/成果；并建立“生成 → 评价 → 复核”的闭环。

- **解析输出（AI 生成）：** 应用场景、通俗特征、关键词摘要（标注“AI 生成，可纠错”）。
- **评价/评分机制：** 用户与后台双通道评价（赞/踩 + 标签理由）；低评分/低置信度进入后台复核池。
- **复核策略：** 运营/技术经理人在后台可编辑解析结果并回写，形成可追溯版本。

#### 6. 聊天与咨询页 (Message)

- **消息列表：** 显示所有对话记录。
- **消息列表操作：** 顶部搜索支持会话对象名称/机构名与最近消息关键词检索；“更多”支持全部标为已读与批量管理（删除/清空历史）。
- **消息列表样式：** 扁平列表（无卡片阴影），头像 + 名称/时间 + 消息摘要；不展示内容类型标签；未读红色角标（可显示数字）；官方/系统会话展示蓝色小型认证标识（圆点/勾）。
- **消息列表空态：** 暂无消息插画 + 文案（标题 + 描述）。
- **会话类型与触发：**
  - **平台客服助手：** 由客服账号（role=cs）或官方客服白名单触发；触发于用户发起咨询、订金支付成功后分配客服、退款/争议介入、材料补充提醒、关键节点跟进。
  - **交易通知：** 由系统账号（role=admin）或系统通知白名单触发；触发于订单创建/支付/合同/退款/结算等状态变更。
- **演示数据（P0）：** 消息列表默认包含“平台客服助手”“交易通知”两类示例会话，用于展示样式与标识。
- **聊天窗口：**
  - **仅支持发送文字与表情**；不支持在聊天内直接上传图片/文件。
  - **材料引用：** 仅可引用发布时上传的佐证材料（聊天内发送“引用卡片”）。
  - **会话空态：** 暂无会话消息插画 + 文案（标题 + 描述）。
- **合同引用：** 合同在协商后由合同中心上传（仅卖家上传 PDF），聊天内仅引用电子版供双方查阅；合同文件对外展示为“发布方水印版”。
  - **顶部栏：** 显示“与某经理/机构”标题，左侧返回，右侧更多按钮；整体浅色底与内容区同风格。
  - **安全提示条：** 顶部展示“请勿私下转账，谨防诈骗”的警示横幅，带警示图标与关闭入口。
  - **咨询卡片：** 会话开始处展示当前沟通的专利/书画/需求信息卡（缩略图 + 标题 + 标签/价格 + “查看详情”）。
  - **对话区样式：** 浅灰背景；双方气泡样式区分（对方白/浅灰、我方绿色），圆角更大；头像随消息显示（对方左、我方右）。
  - **时间与分隔：** 时间居中弱化展示，气泡下方或间隔显示，消息间距更清晰。
  - **输入栏：** 底部圆角输入框；右侧提供“表情/引用”入口；“发送”按钮小圆形橙色。

    - **待确认：** 双方查阅水印版合同并确认；可提醒对方确认。
    - **可查阅：** 合同已确认，支持复制链接/查看；聊天内仅引用合同电子版。
- **发票管理中心（P1 规划）：** 入口在「我的-发票中心」，支持 **待申请 / 申请中 / 可下载** 三种状态。
  - **待申请：** 已完成订单未开票，可提交开票申请。
  - **申请中：** 财务审核/开票处理中，可查看进度。
  - **可下载：** 发票已开具，支持小程序内直接下载（不再复制链接）。
  - **订单详情联动：** 订单详情页支持跳转至发票中心并定位当前订单。
  - **P0 说明：** 仅保留“财务上传/下载发票附件”的能力；申请/流转态仅作 UI 演示与 P1 预留。
  - **发票中心空态：** 暂无发票插画 + 文案（标题 + 描述）。

  - **入口：**「我的-交易规则」。
  - **内容结构：** 平台撮合定位说明、订金与尾款支付规则、退款/争议处理原则、线下合同与权属变更说明、风险提示与免责声明。

- **入口：** 首页快捷入口“发布”/悬浮按钮/「我的-发布管理」。
- **发布类型：** 专利交易（商品）、书画专区（作品交易）、产学研需求、成果展示。
- **专利交易发布（建议字段）：**
  - 基础：专利号/申请号/公开号（手动填写）→ 可选填写名称、类型、分类号、申请日等。
  - 权属：当前权利人/申请人信息、权属证明材料上传（证书/合同链路等）、是否质押/许可现状声明。
  - 交易：交易方式（转让/许可）、标价/面议、订金、可谈空间、可交付资料清单、预计周期。
  - 风险提示：平台仅撮合，最终以合同及主管部门登记结果为准。
  - 交互与样式（对齐目标设计）：
    - 白底导航栏（返回 + 居中标题“发布专利”）；页面底色浅灰。
    - 表单容器为扁平白卡（圆角 16rpx、弱描边、无阴影）。
    - 输入/下拉统一为浅灰底、圆角 12rpx、高度约 72rpx；placeholder 更浅且统一小字号。
    - 字段标题（输入框外标签）统一黑色加粗，突出层级。
    - 关键字段行采用两列布局：专利类型 + 交易方式；IPC + LOC；其余单列纵向排列。
    - 权属材料使用虚线边框上传框（居中图标+说明文字）。
    - 底部操作按钮：左侧描边“保存草稿”，右侧实心“提交审核”。
- **书画专区发布（建议字段）：**
  - 基础：作品名称、作者/创作者、创作日期、类别（书法/绘画）、书体/题材、尺寸/材质。
  - 权属：著作权登记证书编号、权属证明材料上传（证书/授权链路）。
  - 展示：作品介绍、图片（≥1 张，可选视频/附件）。
  - 交易：标价/面议、订金、地区、可交付资料清单。
  - 外部检索：填写证书编号后支持“检索/核验”（P0 先走后台人工核验，P1 接入数据源）。
- **需求发布（建议字段）：** 需求标题、技术领域/关键词、预算范围、合作方式（专利转让/技术咨询/委托开发/平台共建）、期望地区/行业、联系人（脱敏展示）。
- **成果展示发布（建议字段）：** 成果标题、图片/视频、应用场景、成熟度、所属单位、可合作方式。
- **审核：** 提交后进入“待审核”池；审核通过上架，驳回需给出原因并支持修改后再提交。

### 第二部分：管理后台（PC Web 端）功能明细

这是运营人员使用的核心系统。

#### 1. 工作台 / 数据看板 (Dashboard)

- **核心数据：** 今日新增用户、今日发布信息数（专利/书画/需求/成果）、今日订单数、今日流水（订金/尾款）、待结算金额、已结算佣金。
- **待办事项提醒：** 待审核发布 X 条、待审核认证 X 条、**新增待介入交易 X 单**。

#### 2. 专利地图数据管理 (Map Data CMS)

- **功能目标：** 解决手动上传数据的需求。
- **地图类型（P1 预留）：** 专利地图（发明/实用新型/外观设计）、书画成果地图、技术经理人地图、科学家地图。
- **操作界面：**
  - **区域管理：** 添加/编辑省份或城市节点。
  - **数据录入：** 针对每个区域，录入以下字段：
    - 年份。
    - 专利总量。
    - 产业分布数据（例如：输入“电子信息: 500条”, “生物医药: 300条”）。
    - 重点企业名单及专利数。
  - **支持 Excel 导入：** 提供模板下载，批量导入数据生成地图。
  - **地域产业标签管理（新增）：** 为省/市配置“特色产业标签”（用于推荐与搜索加权）。
  - **特色产业专利标记（新增）：** 在省/市维度维护“省级/市级特色产业专利”名单，配置置顶优先级/权重，突出区域特色。

#### 3. 平台内容库管理 (Platform Content CMS)（P1 预留）

- **功能目标：** 后台维护平台自有专利/书画/需求/成果数据（用于冷启动与重点展示）。
- **操作界面：**
  - **内容上传/导入：** 单条录入 + Excel/批量导入。
  - **来源标识：** `source=PLATFORM`（默认免审但保留下架/回滚）。
  - **版本记录：** 关键字段修改留痕，支持历史回溯。

#### 4. 交易订单管理 (Order System)

- **列表展示：** 订单号、专利/书画作品名、买卖双方、交易金额、当前状态、**跟单客服**。
- **状态流转操作（关键）：**
  - **状态：已付订金。** -> 操作：**分配客服**（系统可自动或手动指派）。
  - **状态：线下合同签署中。** -> 客服在后台记录里程碑（合同已寄送/已签署/回传材料齐全等）。
  - **状态：待付尾款。** -> 里程碑“合同签署确认”完成后，系统通知买家支付尾款（线上支付）。
  - **状态：尾款已付（托管）/权属变更中。** -> 客服持续记录进度（提交变更/补正/变更完成）。
  - **状态：待结算/放款。** -> 变更完成并确认无争议后，操作“确认结算并放款”（系统扣除佣金，向卖家放款）。
  - **退款处理：** 支持订金退款/尾款退款（按 7.1 规则自动或人工审批），全链路留痕与对账。

#### 5. 信息发布审核 (Content Audit)

- **审核列表：** 每一条用户发布的信息（专利、书画、需求、成果）都需要先进入“待审核”池。
- **审核详情：** 查看图片是否违规、文字是否含敏感词、专利号是否格式正确；专利类型/分类号是否合理；书画证书编号是否完整；权属材料是否齐全（如适用）。
- **操作：** 通过（上架） / 驳回（填写驳回原因，用户端会收到通知）。

#### 6. 用户与认证管理 (User & Auth)

- **用户列表：** 查看所有注册用户，支持封禁/解封。
- **认证审核：**
  - 查看用户上传的身份证/营业执照/事业单位法人证书等材料。
  - 人工核对信息是否一致，点击“通过认证”或“驳回”。
  - 认证通过后，前端显示“V”标或“企业认证”标签。
- **技术经理人栏目管理：** 列表展示开关、排序/推荐配置、服务标签维护（与审核通过状态联动）。

#### 7. 财务管理 (Finance)

- **资金流水：** 查看订金/尾款入账、退款、放款、佣金入账等流水记录。
- **结算与放款：** 订单结算审核（如需）、卖家放款记录与失败重试。
- **发票管理（新增）：** 已完成订单发票上传/替换（仅 PDF）+ 发票号码/日期（可选）+ 通知下载 + 审计留痕。
- **报表导出：** 导出月度交易报表，统计成交额、佣金收入、退款率、放款成功率。

#### 8. AI 解析与匹配运营（P1 预留）

- **解析结果复核：** 查看置信度、错误样本，支持人工编辑并回写。
- **评价统计：** 用户/后台评分趋势、低分原因标签。
- **匹配策略：** 关键词拆解规则、行业词表/热词库维护。

#### 9. 专利托管监控（P1 预留）

- **年费日程表：** 按专利类型/申请日自动生成年费提醒。
- **工单跟踪：** 到期提醒 → 客服线下托管 → 回填处理结果与凭证（后台闭环）。
- **展示口径：** 仅后台展示，不在小程序流程内呈现。

#### 10. 告警中心（P1 预留）

- **渠道：** 短信/邮件/站内消息（支持模板化）。
- **触发：** 年费到期、AI 复核积压、导入失败、支付回调异常、审核积压等。
- **策略：** 去重/冷却/升级规则。

#### 11. 系统设置

- **Banner 配置：** 上传首页轮播图图片和跳转链接。
- **协议管理：** 编辑《用户协议》、《交易服务条款》的富文本内容。
- **客服设置：** 配置前端显示的客服电话、默认自动回复语。
- **类目与标签：** 配置行业领域、分类号映射、热门搜索词、敏感词库；维护书画专区的类别/书体/题材/材质字典。
- **推荐配置（新增）：** 调节推荐权重系数（发布时间/点击/收藏/咨询/地域/特色置顶/兴趣标签）、时间衰减参数与刷量去重策略；权重调整后立即影响推荐排序。
- **书画外部检索配置（P1 预留）：** 著作权登记检索数据源、接口密钥、检索频率与缓存策略。
- **AI/语音配置（P1 预留）：** ASR 供应商、热词库、解析模板/提示词、置信度阈值。
- **告警配置（P1 预留）：** 告警渠道、接收人、阈值与升级规则。

#### 12. 大数据分析中心（P1 预留）

- **交易库/成果库/产学研库：** 交易额、转化率、需求匹配、成交周期等指标看板。
- **报表能力：** 支持按时间/区域/行业维度统计与导出。
- **数据来源：** 订单/需求/成果/咨询/搜索日志（P1 规划，先预留架构与入口）。
