# 系统功能测试计划（P0 全量）

> 目标：覆盖上线前 **P0 主链路** 的端到端功能验证（含异常/边界），作为上线前的统一测试口径。

## 1. 范围与入口

- 范围：`apps/client`（小程序/H5）、`apps/admin-web`（后台）、`apps/api`（服务端）
- 依赖：真实/联调环境 + 真实支付/登录接入后执行；演示环境可先跑 Mock/手工确认链路
- 口径：P0 必测；P1 仅做冒烟

## 2. 测试环境与数据准备

- 环境：
  - Dev：演示/Mock（允许 demo token/手工确认）
  - Staging：真实登录/支付/回调接入后的联调环境
- 账号：
  - 买家、卖家、后台运营/客服/财务/管理员
- 测试数据：
  - 上架内容（专利/需求/成果/书画）各 1 条
  - 订单至少 2 笔（含一次正常完成、一次退款）
  - 发票至少 1 次完整闭环（申请→上传→下载）
  - 文件上传/下载至少 2 种（合同/发票）

## 2.1 非 Mock 实测（Dev / 本地）

> 目标：在不接入真实微信登录/支付前，验证真实 API + 数据库链路与基础状态机可用。

- 前置：docker compose 基础设施已启动（postgres/redis/minio）
- 执行：
  - `powershell -File scripts/api-real-smoke.ps1 -ApiPort 3248`
  - `powershell -File scripts/db-preflight-check.ps1`
  - （可选）WeApp 路由冒烟（无截图，DevTools 自动化）：`node scripts/weapp-route-smoke.js --cli-path <cli.bat> --project-path apps/client --user-token <DEMO_USER_TOKEN>`
  - 小程序手工冒烟：首页/搜索/详情/消息/收藏/个人中心/发布（清单：`docs/engineering/weapp-manual-smoke-checklist.md`）
- 产出：将结果记录到 `docs/engineering/test-report.md`

## 4. 异常与边界用例（必须）

- 重复下单/重复支付意图：幂等键返回已完成或处理中结果
- 退款重复提交：幂等返回，不产生重复退款单
- 无权限访问：订单/发票/工单/文件全部拒绝
- 回调重放：支付/退款回调重复请求不改变状态
- 失败兜底：支付失败/回调失败/上传失败的重试与提示

## 5. 执行记录与输出

- 执行记录：写入 `docs/engineering/test-report.md`（新增 `Latest (YYYY-MM-DD)` 区块）
- 结论要求：
  - P0 全部通过才可上线
  - P1 可延后但需记录风险与处理计划
