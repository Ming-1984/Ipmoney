# Ipmoney 平台功能与架构说明书

## 1. 平台定位与业务价值

Ipmoney 是面向知识产权交易与运营的数字化平台，围绕“标的发现、交易协商、履约交付、资金结算、运营治理”构建统一业务闭环，目标是将分散的交易活动沉淀为可标准化、可复制、可审计的平台能力。

平台价值主要体现在三层：

- 经营价值：形成可持续扩展的平台化经营模式，降低组织对个体经验的依赖
- 用户价值：提升交易透明度与履约确定性，降低信息不对称和沟通成本
- 管理价值：实现跨角色协同作业与全过程留痕，支撑风险控制与经营复盘

## 2. 核心能力全景

| 能力域 | 业务目标 | 核心能力 |
|---|---|---|
| 供需撮合 | 提升有效线索与成交效率 | 搜索检索、标签推荐、专题运营、咨询触达、供给发布 |
| 交易履约 | 保证交易过程可执行、可追踪 | 订单、订金/尾款、合同、发票、退款、结算、状态机管理 |
| 服务协同 | 提升复杂问题处理效率 | 会话中心、工单流转、证据留存、SLA 跟踪、跨角色协作 |
| 运营治理 | 降低运营成本并提升一致性 | 审核中台、批量运营、配置中心、任务看板、策略联动 |
| 风控合规 | 控制资金与流程风险 | RBAC、审计日志、幂等机制、告警分级、异常闭环 |
| 增值服务 | 形成长期收入与客户黏性 | 年费托管、维保计划、执行任务、回执对账、过程归档 |

## 3. 功能体系

### 3.1 用户端功能体系

- 发现与检索：支持挂牌、成果、机构、发明人、技术经理等多维检索
- 咨询与沟通：从详情页一键发起咨询，支持多轮会话与消息沉淀
- 发布与管理：支持专利交易/成果发布、认领申请、进度追踪
- 交易与履约：覆盖下单、分阶段支付、合同上传、发票查看、退款申请
- 个人中心：统一管理认证、资料、地址、通知、客服与年费托管入口

### 3.2 管理端功能体系

- 运营域：认证审核、挂牌审核、专利导入、批量上架、首页运营位配置
- 客服域：会话接待、工单升级、争议处理、SLA 与处理时效追踪
- 财务域：退款审核、结算放款、发票归档、报表导出
- 治理域：账号开通、角色权限、审计追溯、告警处置

### 3.3 关键业务闭环

- 交易主链路：咨询 -> 下单 -> 订金 -> 合同 -> 尾款 -> 变更 -> 结算 -> 发票
- 退款闭环：申请 -> 审核 -> 执行 -> 状态回写 -> 记录沉淀
- 争议闭环：会话识别 -> 工单升级 -> 证据上传 -> 处理流转 -> 闭环复盘
- 年费托管闭环：计划创建 -> 任务执行 -> 托管订单 -> 回执对账 -> 完成归档

## 4. 平台优势

### 4.1 全链路一体化优势

- 从线索获取到资金结算形成统一链路，减少跨系统断点
- 关键状态统一管理，提升流程确定性与协同效率
- 交易、服务、财务数据可关联，支持端到端复盘

### 4.2 标准化作业优势

- 后台按“查询 -> 处理 -> 留痕 -> 复核”模型设计，降低作业波动
- 运营、客服、财务、管理员职责边界清晰，协作成本可控
- 配置中心支持策略参数化，业务规则可快速迭代

### 4.3 技术与扩展性优势

- 统一 API 治理口径：鉴权、权限、日志、审计统一落地
- 模块化架构按业务域解耦，便于独立演进与平滑扩展
- 小程序/H5 同构接入，兼顾触达效率与实施成本

### 4.4 风控与合规优势

- 身份门禁 + 权限门禁双重控制关键操作
- 支付、退款、发票等关键接口支持幂等，降低重复执行风险
- 关键操作前后快照可审计，满足追责与合规检查要求

## 5. 总体架构

### 5.1 架构分层

- 终端接入层：微信小程序为主，H5 为补充入口
- 业务协同层：用户端与管理端统一接入业务服务
- 业务服务层：NestJS 模块化 API，按交易、服务、治理等域组织
- 数据基础层：PostgreSQL、Redis、对象存储、审计日志、告警能力

### 5.2 领域模块

- 交易域：订单、支付、合同、结算、发票、退款
- 供给域：专利、挂牌、成果、认领、发布管理
- 服务域：会话、消息、工单、客服协同
- 治理域：RBAC、配置中心、审计日志、告警中心
- 增值域：年费托管计划、任务、订单、时间线

### 5.3 设计原则

- 一致性：统一鉴权、统一权限、统一日志、统一审计
- 最小权限：按角色最小授权，防止长期越权
- 状态可控：关键对象采用状态机管理，确保流程可验证
- 可演进：模块化拆分，支持后续能力增量接入

## 6. 典型应用场景

### 6.1 场景一：专利交易全流程

- 用户检索标的并发起咨询
- 平台促成下单并进入订金与合同阶段
- 履约完成后进入尾款、结算、开票和归档

业务价值：提升成交效率，降低履约不确定性。

### 6.2 场景二：专利成果发布与认领

- 用户发布成果，平台审核后上线
- 认领申请进入审核并回写归属

业务价值：提升供给质量与可运营性，沉淀结构化资产。

### 6.3 场景三：年费托管服务

- 建立维保计划并自动分配任务
- 通过托管订单推进执行、回执与对账闭环

业务价值：形成持续服务收入，提升客户长期留存。

### 6.4 场景四：争议与退款处理

- 会话识别争议并快速升级工单
- 处理过程保留证据、备注与状态轨迹

业务价值：提升风险处置效率，降低投诉与纠纷成本。

## 7. 应用成效与指标口径

建议按月跟踪以下指标，建立统一经营看板：

- 增长指标：浏览量、咨询量、发布量、活跃用户数
- 转化指标：咨询转单率、下单率、订金支付率、尾款支付率
- 履约指标：合同确认率、变更完成率、结算完成率、平均履约周期
- 财务指标：成交额、佣金收入、退款率、放款成功率
- 服务指标：会话响应时长、工单关闭时长、SLA 达成率
- 风控指标：高风险操作数、告警闭环时长、审计覆盖率

## 8. 平台能力边界与适用范围

平台当前重点能力覆盖如下：

- 交易全流程闭环能力：从咨询、下单、支付、合同到结算、发票，形成可追踪的业务主链路
- 多角色协同作业能力：运营、客服、财务、管理员在统一作业体系中协同处理同一业务对象
- 风险控制与审计能力：关键操作具备权限校验、幂等保护、审计留痕与告警处置机制
- 增值服务承载能力：支持年费托管计划、任务执行、回执上传与对账归档的完整流程

平台适用范围如下：

- 适用于知识产权交易撮合与履约交付场景
- 适用于供给侧发布、审核、认领、运营的标准化管理场景
- 适用于客服争议处理、工单流转、跨角色问题闭环场景
- 适用于对资金流程可追踪、可审计、可复盘有要求的运营场景

平台能力边界说明如下：

- H5 端在支付环节采用引导至小程序完成支付的协同策略
- 资金相关线下执行动作通过平台凭证、状态与日志实现过程留痕
- 外部生态能力通过标准接口对接，按业务需要进行能力接入与治理
